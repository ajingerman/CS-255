# CS-255
CS 255 Portfolio

The DriverPass project focused on designing a system for a client that provides driver training services. The goal was to create a web based system that helps customers prepare for the DMV written exam while also managing driving lesson scheduling. In Project One, I developed a business requirements document that defined system functionality, user roles, and both functional and nonfunctional requirements. In Project Two, I built on those requirements by creating system design diagrams and technical requirements that show how the system would operate. The system includes account creation, DMV aligned training materials, practice exams, package tracking, and the ability to schedule, reschedule, and cancel lessons, along with support for staff roles such as the secretary, IT officer, and system owner. One thing I did well was making sure the requirements were detailed and connected to the client’s needs from the interview, especially with scheduling logic, availability checks, and package tracking. I also improved my Project One document after feedback by making the purpose clearer, expanding the system background, separating goals from objectives, fixing citations, and removing personal references. If I could revise one part of my work, I would improve the Gantt chart by breaking the project into more detailed steps with clear start and end dates, adding a legend to show task groupings, and identifying which roles are responsible for each task so the workflow is easier to understand.

To interpret the user’s needs, I relied on the DriverPass interview and translated those needs into specific system requirements and design decisions. For example, the need for accurate scheduling became system checks for instructor, vehicle, date, and time availability, while the need for progress tracking became requirements for practice test results and customer training records. Considering user needs is important because the system must reflect how customers and staff actually interact with it, or it will not be effective. When designing software, I approach the process by breaking the system into smaller parts and focusing on how each user interacts with it. I used structured requirements, UML diagrams, and technical requirements to organize the system and show how different parts connect. In the future, I will continue using interviews, clear requirements, step by step logic, and system modeling to make sure designs are complete and aligned with user needs.
