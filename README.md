# CS-255
CS 255 Portfolio

The DriverPass project focused on designing a system for a client that provides driver training services. The goal was to create a web-based system that helps customers prepare for the DMV written exam while also managing driving lesson scheduling. The system includes features such as account creation, access to DMV-aligned training materials, practice exams, and the ability to schedule, reschedule, and cancel lessons. It also supports staff roles, including the secretary, IT officer, and system owner, allowing them to manage customer records, control accounts, and generate reports.

One thing I did well in this project was making sure the system requirements were detailed and directly connected to the client’s needs from the interview. I clearly defined how scheduling works by including checks for instructor and vehicle availability, preventing overlapping reservations, and tracking lesson usage within customer packages. I also made sure that user roles were clearly defined so that each type of user could only access the parts of the system relevant to them. Another area I improved on was taking the feedback from Project One and revising the document to better match the assignment requirements. This included fixing issues with formatting, removing personal references, improving citations, and making sure each section fully answered the questions that were asked.

If I could revise one part of my work, I would improve the Gantt chart. I did not fully align it with all of the system requirements and the development process described in the project. If I had more time, I would break the work into more detailed steps and make sure each task clearly connects to the system features, such as account management, scheduling, training content, and reporting.

To interpret the user’s needs, I relied heavily on the DriverPass interview and translated each requirement into specific system behavior. For example, I turned the need for accurate scheduling into system checks that verify instructor, vehicle, date, and time availability before confirming a reservation. I also included requirements for tracking customer packages and recording test results so that the system reflects how the business actually operates. Considering the user’s needs is important because the system must match how customers and staff interact with it. If those needs are not clearly understood, the system can become difficult to use or fail to support the business.

When designing software, I approach it by breaking the system into smaller parts and focusing on how each part functions. I used structured requirements and system design diagrams to show how data moves through the system and how users interact with it. I also focused on validation, such as preventing duplicate accounts and ensuring required fields are completed, to maintain accuracy. In the future, I will continue using clear requirements, step-by-step logic, and system modeling to make sure designs are complete and aligned with user needs.
